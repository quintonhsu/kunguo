背景： 离线指标T+1是实时+离线的融合性方案，既依赖于每天增量的离线数据，也依赖于实时scan hbase并现算的过程。

业务场景：对于某个模型的入参(即指标)进行hbase查询，获取结果后再进行计算，得到对应的指标值。具体说来，传入某个id(身份证)和模型里的比如30个指标，采用多线程，查询出每个hbase的结果并计算出对应的指标值。

压测：随机挑选了100个指标，用5000组不同的id/mobile作为入参，查询并计算100个指标的值，看RT和吞吐量是否符合预期。经过压测后，发现平均RT居然需要230ms，即使每个指标都会scan数条hbase的rowkey, 那也不至于啊，不满足要求。

性能优化了多处地方，感谢黄波和昌龙对我在优化过程中的一些支持和帮助，下面列出了一些提升比较明显的几步。

一、算子库对长名重复解析优化

       在指标平台调用整个算子库的生命周期中，有几处方法调用都会传入长名，使得长名的解析过程反复调用，通过review算子库代码，调整为只调用一次。再次压测，性能提高了10%。

二、离线指标长名解析提前预热

        离线指标的数量是可枚举的，在系统初始化时就加载所有的指标长名，把每个长名转换成hbase所需要的column列表，并做好映射，而不用每次传入长名都去转换，提高2~3毫秒RT。

三、算子库每次都会去处理两个地方，非常耗性能

        指标平台(Java)每次都会调用算子库(Scala)的calculate方法, 在代码review前，calculate的方法每次都会去new SimpleDateFormat对象，并且把一个包含行业信息size>10000的Java List转换成Scala的List, 尤其是后者，100个指标中每个指标调用都会调用这个方法，非常耗时，如下:

业务上也没有必要每次调用，只需要构造函数时初始化一次即可，所以把SimpleDateFormat和Industry行业信息提炼出来：

再次压测，性能突然提高了7倍，主要是得益于Industry行业信息转换方式发生了变化，SimpleDateFormat的提炼只是减少了对象开销，性能上的提升一般。

四、SimpleDateFormat类线程不安全，造成1%的指标计算结果有误

        对指标计算结果进行了对比，发现大概有1%的指标计算结果不太正常，通过日志，发现了许多异常： 在调用format.parse前打印出date的值，业务上每次都会传同一个值，没有问题。最开始还觉得SimpleDateFormat是一个很通用的时间工具类，应该不会不安全吧，后来发现这家伙调用parse的时候会调用calender这个共享变量，致使线程不安全。所以黄波同志作了threadLocal处理，解决之。

五、去掉关键点日志

        压测的时候发现总有个pid非常消耗CPU，有时会消耗掉20~30%的CPU，所以排查之。由于当时具体的pid和tid已忘记，下面仅仅是举个例，主要讲排查的方法和手段。

  • ps -ef grep java 获取当前Java进程号， 比如是12345
  • top -Hp 12345 可查询pid=12345进程中哪个线程占用cpu过高，通过列表查看到消耗cpu最高的tid为24457
  • jstack -l 12345  » stack.log 把12345进程中所有的线程栈信息打印到某个日志文件
  • 把24457转化为16进制，为0x5f89, 在stack.log里搜索5f89关键字，如下

从日志可以看到，是graylog客户端通过socket发送日志到Gelf server，导致cpu消耗较大。由于在预发环境上压测在很多地方加了日志，用于判断每个步骤的耗时，所以产生了huge多的日志，把没必要的日志打印关闭，性能提升20%。

六、扩大dimmension线程，性能提高了40%

        前面的优化步骤都是基于1个并发来进行压测，平均RT和99%RT基本达到要求，但吞吐量不足。于是加大了并发量，模拟5个并发用户请求。之前1个并发时获取指标的ThreadPoolExecutor线程池是100，由于是5个并发，线程池改为500个。

本以为吞吐量立即会上来，但事实不是，经过几轮尝试，不管ThreadPoolExecutor设置100、200、或者500，吞吐量也没有明显提升，即使增大了hbase client的线程池，效果也不明显，排查之。

  • ps -ef grep java 获取当前Java进程号， 12345
  • top -Hp 12345 查询pid=12345进程中所有的线程的cpu占用率都不高，均在0~1%之间，作为一个需要大量计算的系统，这个值显然不正常
  • jstack -l 12345  » jstack.log dump当前线程状态

下图列出了整个业务流程调用路径中经过的主要线程池：

  1. tomcat线程池 Springboot中默认内嵌的是tomcat server, 看了下tomcat源码，最小线程数是10，最大线程数是200，grep了jstack.log中的http-nio线程数，远不到200，也就是压测的力度较小，tomcat还没必要创建更多的线程来处理任务，所以吞吐量的瓶颈不在此线程池。

  2. Call Dim线程池 call dim线程池是用于多线程去处理不同的主维度，比如用户传来一个手机号和一个身份证，那么call dim线程池可以并行处理两个主维度，所以call dim线程数设置为2。通过观察jstack.log, 发现出现了大量调用callDim方法时的WAITTING状态，怀疑瓶颈出现在线程池。

  3. Call Index线程池 call index线程池是用于多线程去处理划分主维度后每个指标的值。线程数设置的是500，通过观察jstack.log, 发现出现了少量调用call方法时的WAITTING状态, 要说是瓶颈应该还好。

  4. Hbase Client线程池 Hbase Client线程池用于查询hbase数据。线程数设置为500，通过观察jstack.log, 如下图，部分线程出现了take队列并且await的状态，说明队列中已经没有任务，线程处在空闲状态。另一方面，hbase的线程池队列是SynchronousQueue，如果workingThread一旦超过500，日志也会报RejectedExecutionException，但日志也正常。所以综合两方面，该线程池也没问题。

综合下来，应该是第二步Call Dim线程池设置过小，导致吞吐量低，所以开大该线程池，cpu利用率明显提升，再次压测性能提升40%

上线结果：

目前离线指标T+1已上线一周，运行较稳定。重点观察了一个合作方的调用，上海六台机器，某个时刻总的TPS为91, rt_p999在20毫秒以内。

调优总结：

1、打印日志适可而止，即使异步写日志，量大的情况还是比较耗资源的

2、循环或者嵌套循环一定要注意大对象的重复生成

3、系统中的线程池设置一定要根据业务场景和cpu数综合判断，设置时要慎重。

4 、jstack里的线程状态要搞清楚，比如什么WATING\TIMED_WATING\BLOCKED的区别，即使是WAITING，也要清楚是什么导致等待，是空闲的线程在等待任务的到来，还是某个线程在等待FutureTask完成后的唤醒，这对于判断线程的真实状态很有帮助。
